CHANGES FOR final01.py

This document records the current specification and implementation of the music timer program and then lists the major changes made while transferring the code into final01.py.

CURRENT FILE DESCRIPTION (based on the current working version, status02.py)

Purpose:
The current file connects to Spotify through the Spotipy API, accepts a Spotify playlist URL or URI from the user, starts playback on the active Spotify device, and stops playback either after a duration or at a specific clock time.

Current implementation summary:
1. Environment variables are loaded from the repository root .env file.
2. Spotify authentication is handled through SpotifyOAuth.
3. The program checks for an active device and falls back to the first available device if none is currently active.
4. The playlist input is converted into a Spotify playlist URI.
5. Playback starts immediately after device and playlist validation.
6. Stop logic is handled in one of two ways:
   - stop after a given number of minutes
   - stop at a given wall-clock time in HH:MM format

Current imported modules:
- os
- pathlib.Path
- dotenv.load_dotenv
- spotipy
- spotipy.oauth2.SpotifyOAuth
- time
- datetime / timedelta

Current variables and names:
- SCOPES: Spotify permission scope string
- sp: Spotify client object
- playlist: raw user input for playlist URL or URI
- playlist_uri: normalized Spotify playlist URI
- device_id: ID of the active or selected playback device
- mode: user-selected stop mode, such as "duration" or "time"
- minutes: number of minutes until playback should stop
- stop_time: user-entered stop time in HH:MM format
- now: current datetime at runtime
- stop_dt: computed datetime for stopping playback
- seconds: number of seconds to wait before stopping
- seconds_to_target: duplicate computation of time-until-stop used for printing

Current functions and I/O:

1. get_spotify_client()
   Input:
   - no direct user input
   - reads SPOTIPY_CLIENT_ID, SPOTIPY_CLIENT_SECRET, and SPOTIPY_REDIRECT_URI from the environment
   Output:
   - returns an authenticated spotipy.Spotify client

2. get_active_device(sp)
   Input:
   - sp: authenticated Spotify client
   Output:
   - returns a Spotify device_id string
   - raises RuntimeError if no devices are available

3. extract_playlist_uri(playlist_url_or_uri)
   Input:
   - a Spotify playlist URL or Spotify playlist URI string
   Output:
   - returns a normalized Spotify playlist URI in the form spotify:playlist:<id>
   - raises ValueError if the input is not recognized as a playlist URL or URI

4. start_playlist(sp, playlist_uri, device_id)
   Input:
   - sp: authenticated Spotify client
   - playlist_uri: Spotify playlist URI
   - device_id: playback device ID
   Output:
   - starts Spotify playback on the target device
   - no explicit return value

5. pause(sp, device_id)
   Input:
   - sp: authenticated Spotify client
   - device_id: playback device ID
   Output:
   - pauses Spotify playback
   - no explicit return value

6. stop_after_duration(sp, device_id, minutes)
   Input:
   - sp: authenticated Spotify client
   - device_id: playback device ID
   - minutes: integer or numeric duration in minutes
   Output:
   - waits for the specified amount of time and then pauses playback
   - no explicit return value

7. stop_at_clock_time(sp, device_id, stop_time_str)
   Input:
   - sp: authenticated Spotify client
   - device_id: playback device ID
   - stop_time_str: string in HH:MM format
   Output:
   - computes the next matching clock time, sleeps until then, and pauses playback
   - prints the seconds until stop and the target stop time
   - no explicit return value

8. main()
   Input:
   - playlist URL or URI from stdin
   - stop mode from stdin
   - either a duration in minutes or a stop time in HH:MM from stdin
   Output:
   - prints connection and status messages
   - starts and later pauses Spotify playback
   - no explicit return value

Current user inputs:
- Playlist URL or URI
- Stop mode: duration or time
- If duration: number of minutes
- If time: stop time in HH:MM 24-hour format

Current user-visible outputs:
- Spotify authentication browser flow
- "Connected to Spotify!"
- Prompt messages for playlist and mode
- A countdown-style message such as "Will stop in X seconds (at HH:MM:SS)."
- Error traces if authentication, input parsing, or device lookup fails

MAJOR STRUCTURAL CHANGES FOR final01.py

1. Clarify program flow in main()
   - Keep authentication, playlist validation, device selection, playback start, and stop scheduling as separate logical stages.
   - Move success messages so they reflect the actual stage completed.
   - Example: print Spotify connection success after authentication, and playlist validation success only after the playlist input has been checked.

2. Add explicit error handling
   - Wrap main control flow in try/except blocks.
   - Catch ValueError for invalid playlist links or invalid time input.
   - Catch RuntimeError for missing devices.
   - Catch generic Exception only as a final fallback.
   - Replace raw tracebacks with clearer user-facing messages.

3. Tighten playlist validation
   - Restrict playlist parsing to actual Spotify playlist links or spotify:playlist: URIs.
   - Avoid accepting unrelated URLs that merely contain playlist-like text.

4. Remove duplicated time calculations
   - seconds and seconds_to_target currently represent the same idea.
   - Compute the delay once and store it in one variable only.

5. Keep environment loading explicit and stable
   - Continue loading the .env file from the repository root using Path(__file__).resolve().parent.parent / ".env".
   - This prevents failures caused by running the script from different working directories.

6. Separate business logic from user interaction
   - Keep input() and print() calls mostly inside main().
   - Let helper functions focus on computation or Spotify API interaction.
   - This makes testing easier and makes later refactoring to a GUI or web interface easier.

7. Make stop-mode handling easier to extend
   - Keep duration and time as separate branches, but organize them so that a future songs mode can be added cleanly.
   - A helper dispatcher or a mode-to-function map would make this cleaner.

CHANGES AND CURRENT SPECIFICATION FOR final03.py

This document summarizes the current program specification and the essential structural decisions up through final03.py. It is intended to serve as a clean version-control reference for the state of the project.

CURRENT PROGRAM PURPOSE

The program connects to Spotify through the Spotipy API, asks the user for a playable Spotify resource, starts or resumes playback on an available Spotify device, and pauses playback either after a selected duration or at a specified clock time.

The program now supports smart playback behavior:
- If the requested Spotify resource is already the current playback target and Spotify is already playing it, the program leaves playback alone.
- If the requested Spotify resource matches the current playback target but playback is paused, the program resumes from the user's current playback position instead of restarting the song or context.
- If nothing is currently playing, or if the user is currently playing something different, the requested Spotify resource starts from the beginning.

SUPPORTED PLAYABLE INPUTS

The program accepts any Spotify resource that can reasonably start playback or create a playable queue:
- playlist links and URIs
- album links and URIs
- artist links and URIs
- track links and URIs
- episode links and URIs

The program rejects:
- non-Spotify links
- malformed Spotify links
- unsupported Spotify resource types
- empty input
- Spotify resources that cannot be validated through the API

ENVIRONMENT AND AUTHENTICATION

The program loads environment variables from a .env file. It first looks in the same directory as the script and then falls back to the parent directory. Required environment variables are:
- SPOTIPY_CLIENT_ID
- SPOTIPY_CLIENT_SECRET
- SPOTIPY_REDIRECT_URI

Spotify authentication is handled through SpotifyOAuth.

MAJOR STRUCTURAL CHANGES ACROSS VERSIONS

1. Environment loading was made explicit and stable.
   - Earlier versions depended on the working directory.
   - Current version resolves the .env path deliberately.

2. Playback input handling was generalized.
   - Earlier versions only accepted playlist links.
   - Current version accepts multiple playable Spotify resource types.

3. Input validation was moved into a prompt loop.
   - The program now reprompts until the user enters a valid playable Spotify resource.

4. Parsing and validation were separated.
   - One layer parses the URL or URI.
   - Another layer confirms the resource exists and is playable through Spotify.

5. Playback logic was separated from validation logic.
   - Validation returns a normalized playback target.
   - Playback functions operate on that target.

6. Current playback inspection was added.
   - The program now checks Spotify's current playback state before deciding how to start playback.
   - This confirms that the API exposes the user's current playback position through progress_ms and item/context metadata.

7. Smart start/resume behavior replaced always-restart behavior.
   - Matching current playback + already playing -> do nothing.
   - Matching current playback + paused -> resume current position.
   - No playback or different playback -> start requested target from beginning.

8. Time scheduling was simplified.
   - Duplicate timing variables from earlier drafts were removed.
   - The program now uses one computed sleep_seconds value.

9. User interaction was kept mostly inside prompt and main functions.
   - Helper functions focus on parsing, Spotify API calls, and scheduling logic.
